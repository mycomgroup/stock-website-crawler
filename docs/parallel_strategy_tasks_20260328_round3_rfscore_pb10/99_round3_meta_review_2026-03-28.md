## 三、当前最关键的逻辑冲突

### 1. 当前市场状态冲突

- `result_02` 认为当前市场宽度约 **23%**，趋势向下，建议减仓或观望。
- `result_05` 认为当前市场宽度 **38.5%**，趋势向上，建议满仓 20 只。

这不是小误差，而是**会直接导致相反操作**的冲突。

在没有统一口径前，处理原则应为：
- **优先相信更保守的口径**
- **不要让 05 的“满仓结论”覆盖 02 的弱势判断**

### 2. 持仓数规则冲突

- `result_03` 的实测结论是：**15 只最优**，极端市场允许 **0 仓**，绝不放宽到 `Score>=6`。
- `result_05` 的设计结论是：**20/15/10/5** 四档，不设空仓线。
- 正式代码仍是 **20/10** 两档。

这三套规则不能同时成立。

当前应优先采用：
- **03 的规则优先级高于 05**
- 原因：03 更接近实测结论，05 仍是待验证设计稿

### 3. 当前候选数量冲突

- `result_06` 写当前候选 **4只**
- `result_10` 写当前候选 **1只**

这说明至少有一个环节在以下几项上不一致：
- 股票池口径
- 过滤条件
- 是否含异常值剔除
- 是否含停牌/不可买过滤
- 数据时点

在这个问题统一前，不应把“当前可持有 10-20 只”当作已确认事实。

### 4. 研究结论与正式代码尚未对齐

正式策略文件 `/Users/fengzhi/Downloads/git/testlixingren/strategies/rfscore7_pb10_final.py` 里仍存在以下与第三轮主结论不一致之处：

- 仍使用 `base_hold_num = 20`、`reduced_hold_num = 10`
- 次级池仍允许 `RFScore >= 6`
- 候选不足时仍有 fallback 补位逻辑
- 尚未看到 `ROA > 0.5%`、`PE < 100` 的硬过滤落地
- 尚未看到行业上限等约束落地

结论：
- **研究结论已经比代码更先进**
- **当前真正的风险不是“没结论”，而是“结论很多，但正式代码还没统一升级”**

## 四、现阶段最可靠的总结论

### 1. 主线没有变

`RFScore PB10` 仍然是当前全仓库里最值得继续推进的主线。

### 2. 但它不是“原样直接上”

真正可靠的版本，不应理解为：
- “PB10 + 满仓 20只 + 候选不够就随便补”

而应理解为：
- **PB10 主池**
- **PB20 次级池**
- **保持 RFScore=7，不轻易降到 6**
- **候选不足时宁愿减仓/留现金**
- **加入异常值硬过滤**
- **再决定是否上行业约束**

### 3. 当前市场更像“可跟踪、不可激进放大”

从更保守且一致的证据看：
- 当前市场不是理想进攻窗口
- 候选数量偏少
- 行业集中偏高
- 直接放大仓位不稳

所以更稳的态度应是：
- **把 RFScore PB10 当作正式核心框架**
- **但在统一口径和代码前，不把它当成已经完全封板的实盘终版**

### 4. 非主线里最值得保留的两条

- **红利小盘**：最值得保留的股票防守线
- **国债固收+**：最值得保留的低波动防守底仓

ETF、ML、行业增强都还可以留档，但现在不是最优先线。

## 五、最终执行口径

如果今天必须只保留一个最稳的研究结论，我建议写成下面这句话：

> **RFScore PB10 仍是当前最强主线，但正式可执行版本应是“PB10 主池 + PB20 备用池 + RFScore7 不降级 + 候选不足时允许减仓/留现金 + 先补齐异常过滤与口径统一”，而不是直接把现有代码当作最终实盘版本。**
