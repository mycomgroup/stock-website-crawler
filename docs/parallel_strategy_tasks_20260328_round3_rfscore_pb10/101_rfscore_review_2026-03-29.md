# RFScore Review 2026-03-29

## 结论先行

- **没有发现足以替换当前主线的新 alpha 逻辑。**
- 现阶段最值得升级的不是主策略选股逻辑，而是**生产、验证、监控三端的统一口径**。
- 因此本次 review 的升级点是：
  - 把“工具脚本与正式策略完全同构”从建议项升级为 **P0 阻断项**

## 发现 1：监控脚本仍在使用宽口径候选

### `rfscore_quality_monitor.py`

- 当前候选逻辑仍是 `RFScore >= 7 and pb_group <= 2`
- 且没有接入正式版硬过滤和 `buyable` 过滤

影响：
- 质量监控看到的“候选股”可能不是正式策略真正会持有的股票

### `rfscore_monitor_full.py`

- 同样仍在使用 `RFScore >= 7 and pb_group <= 2`

影响：
- 监控报表和真实持仓可能长期不一致

## 发现 2：候选验证脚本仍停留在研究口径

### `validate-rfscore-candidates.js`

- 目前更像“原始研究检查工具”
- 会打印 `PB10` 和 `PB20` 候选，但没有正式版最终候选概念
- 没有接入：
  - `PE < 100`
  - `ROA > 0.5`
  - `buyable` 过滤
  - 行业上限裁剪

影响：
- 它适合研究，不适合作为上线核对工具直接使用

## 发现 3：行业分析脚本仍是原始 PB10 候选视角

### `rfscore_sector_analysis.py`

- 目前分析的是“PB10 原始候选”
- 不是“正式版最终可买候选”

影响：
- 行业暴露分析可能高估或误判实际持仓风险

## 发现 4：实验脚本仍大量沿用 20 只持仓

### 代表文件

- `tmp/rfscore7_variant_eval.py`
- `tmp/rfscore7_refine_eval.py`
- `tmp/test_rfscore_pb10.py`

影响：
- 容易把“20只持仓实验结果”误读为“正式版结论”
- 这会让后续研究继续带着旧偏置走

## 最终判断

### 没有升级主策略 alpha

我没有看到以下这些方向能稳定打赢当前正式版：

- `PB20` 回归主版本
- `PB30-40` 替代 `PB10`
- `Gross Profitability` 二次排序替代现有排序
- `Turnover / CGO` 直接进入 V1
- 更激进或更复杂的 MA60 / RSRS 状态切换

### 升级生产一致性

这次 review 唯一值得立刻吸收进正式方案的升级是：

> **所有 RFScore 的候选验证、质量监控、行业分析、日常快照脚本，都必须直接复用正式版选股逻辑，至少要同构到“主池/次级池/硬过滤/buyable/行业上限/持仓目标”这一层。**

否则上线之后，最容易出问题的不是策略收益，而是：

- 你看到的报表不是策略实际在跑的东西
- 你以为候选有 6 只，实盘其实只剩 3 只
- 你以为行业集中度 30%，实际可买候选已经超过 50%

## 一句话结论

**没有发现新的 alpha 升级点；有一个必须吸收的生产升级点：把“研究脚本口径”和“正式版策略口径”彻底统一，并把它提升为上线阻断项。**
