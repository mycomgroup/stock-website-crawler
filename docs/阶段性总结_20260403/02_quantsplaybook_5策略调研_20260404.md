## P1: 56_mlt_tsmom — MLT-TSMOM 多标的时序动量

### 代码质量评估

**优点**
- 核心算法实现完整：指数衰减波动率 + 多时间窗口信号加权，与原论文逻辑一致
- 单ETF权重上限40%的集中度控制是合理的
- 月频调仓，交易成本可控

**问题与风险**

1. **标的池过窄**：仅4个ETF（黄金、纳指、创业板、沪深300），样本量极小，夏普高可能是过拟合
2. **只做多，丢失了TSMOM的核心**：原始TSMOM是 `sign(r) * vol_scaled`，负信号应做空或空仓，当前实现把负权重直接截断为0，等于只保留了动量的多头部分，策略退化为"动量选ETF"
3. **权重归一化逻辑有缺陷**：`total_w = sum(long_weights.values())` 后再 `w / total_w`，当只有1个ETF有正信号时会100%集中，与40%上限逻辑冲突
4. **缺乏空仓机制**：全部信号为负时仍会持仓（因为截断为0后 total_w=0 才会跳过），但代码里 `if total_w > 0` 的判断会导致全负信号时空仓，这部分逻辑是对的，但没有明确注释
5. **vol_window=60 硬编码**：没有参数化，不便于后续调优
6. **print 语句留在生产代码里**：每月打印一次权重，在平台上会产生大量日志

### 后续改进方向

- **优先级高**：扩展标的池，加入债券ETF（511010）、商品ETF（159980）、海外REITs等，至少8-10个标的，才能体现"跨资产"的分散效果
- **优先级高**：实现完整的多空版本，负信号时减仓到0（已实现）或做空（需要融券），至少要明确空仓逻辑
- **优先级中**：加入波动率目标调整总仓位（Portfolio-level vol targeting），而不只是单标的权重
- **优先级中**：回测区间要包含2015-2016熊市、2018年、2022年，验证跨周期稳定性
- **优先级低**：参数化 lookbacks 和 vol_window，做敏感性分析

## P2: 45_salience_str_factor — 凸显理论STR反转因子

### 代码质量评估

**优点**
- 逻辑简洁，STR因子定义清晰（过去N日最大单日涨幅）
- 选低STR做多（反转逻辑）方向正确
- 月频调仓，等权持仓30只，实现规范

**问题与风险**

1. **因子定义与原论文有偏差**：原论文STR是基于"周收益率"的凸显性，当前实现用日收益率的最大值，两者在统计特性上有差异，日线版本噪音更大
2. **window=52 的含义混乱**：注释说"约一年的周数，日线用52天"，但52天只是约2.5个月，不是一年。如果要对齐原论文的52周，应该用约252个交易日
3. **没有行业中性化**：STR因子在不同行业间差异很大，不做行业中性化会导致持仓集中在特定行业（如高弹性的科技/小盘），回撤会比预期大
4. **沪深300成分股池**：300只股票里选30只（10%），集中度偏高，且沪深300本身偏大盘，STR反转效应在大盘股上可能弱于中小盘
5. **缺乏市值中性化**：低STR股票可能系统性偏向某个市值区间

### 后续改进方向

- **优先级高**：修正 window 参数，用252日对齐原论文的52周，或明确说明是"短期版本"
- **优先级高**：加入行业中性化，在每个行业内分别选低STR股票，避免行业集中
- **优先级中**：扩大股票池到中证500或全A，STR反转效应在中小盘更显著
- **优先级中**：与RFScore结合做增量验证（已在总览中列为P2探索项），STR作为RFScore的补充过滤层
- **优先级低**：测试做空高STR股票的多空版本，验证因子双向有效性

## P3: 42_enterprise_lifecycle — 企业生命周期选股

### 代码质量评估

**优点**
- 季度调仓频率合理，换手率低
- 代码结构清晰，异常处理完整

**问题与风险**

1. **核心逻辑是代理变量，与原论文严重偏离**：原论文企业生命周期是基于现金流特征（经营CF、投资CF、融资CF的正负组合）来判断成长期/成熟期/衰退期。当前实现完全没有用财务数据，而是用价格趋势+成交量趋势+收益率变异系数来"近似"，这是完全不同的因子
2. **RiceQuant 实际上可以获取财务数据**：`get_fundamentals()` 可以拿到现金流数据，没有必要用代理变量
3. **变异系数归一化逻辑有问题**：`ret_stability = -min(ret_cv, 5) / 5` 把变异系数截断到[-1,0]，但这个归一化是任意的，没有理论依据
4. **综合得分权重（0.4/0.3/0.3）是拍脑袋的**：没有任何理论或实证依据
5. **季度判断逻辑有bug**：`current_quarter = (context.now.month - 1) // 3` 在月初可能触发多次（如果handle_bar在季度第一个月的多个bar被调用），应该用 `(year, quarter)` 组合来去重

### 后续改进方向

- **优先级极高**：重写核心逻辑，使用真实财务数据（现金流三分类）实现原论文的生命周期判断，这是让策略有意义的前提
- **优先级高**：修复季度去重bug，改为 `(context.now.year, current_quarter)` 组合判断
- **优先级高**：验证 RiceQuant 财务数据的可得性和时效性（避免未来数据泄露，财报发布有滞后）
- **优先级中**：加入财报发布时间对齐，使用"已公布"的最新财报数据，而不是当期数据
- **优先级低**：与RFScore结合，生命周期作为质量过滤层

## P4: 44_pure_volatility_factor — 纯真波动率因子（IVOL）

### 代码质量评估

**优点**
- 算法实现最规范：OLS回归剔除市场因子，计算特质波动率，与学术文献完全一致
- 代码结构清晰，异常处理完整
- 低波动率异象是学术界验证最充分的因子之一，理论基础扎实

**问题与风险**

1. **只用市场单因子回归**：原论文"纯真波动率"的核心是剔除"跨期截面相关性"，需要用Fama-French多因子（市场、规模、价值）做回归，只用市场单因子得到的是普通IVOL，不是"纯真"版本
2. **window=60 可能不够**：60个交易日约3个月，对于OLS回归来说样本量偏少，beta估计不稳定，建议用120-252日
3. **没有行业中性化**：低IVOL股票系统性偏向防御性行业（银行、公用事业），不做行业中性化会导致行业集中
4. **市场指数用沪深300**：对于全A选股，用沪深300作为市场代理可能不够准确，中小盘股与沪深300的相关性较低
5. **月频调仓但用60日窗口**：信号更新频率与持仓调整频率匹配，但60日窗口意味着信号有较大滞后

### 后续改进方向

- **优先级高**：扩展为Fama-French三因子回归（需要获取SMB、HML因子数据），实现真正的"纯真"特质波动率
- **优先级高**：加入行业中性化，在行业内排序选低IVOL股票
- **优先级中**：将回归窗口扩展到120或252日，提高beta估计稳定性
- **优先级中**：测试不同股票池（沪深300 vs 中证500 vs 中证800），低波动率异象在哪个池子最显著
- **优先级低**：与RFScore结合，IVOL作为风险控制过滤层（剔除高特质波动率股票）

## P5: 35_industry_rotation_pv — 行业量价轮动

### 代码质量评估

**优点**
- 量价动量因子定义合理：价格动量 × (1 + 成交量趋势相关系数)
- 持仓3个行业ETF，分散度适中
- 代码简洁，逻辑清晰

**问题与风险**

1. **行业ETF池覆盖不全且有重叠**：当前10个ETF中，`512010` 和 `512660` 都是军工ETF，存在重叠。同时缺少重要行业：半导体（512480）、白酒（512690）、地产（512200）、券商（512880）等
2. **window=20 太短**：20个交易日约1个月，动量信号噪音很大，容易追涨杀跌。行业轮动通常用3-6个月的动量窗口
3. **没有止损机制**：最大回撤21.79%，用户备注"需加止损/止损"，当前代码完全没有止损逻辑
4. **量价因子权重固定**：`price_mom * (1 + vol_trend)` 中成交量趋势的权重是隐式的，没有参数化
5. **成交量趋势用线性相关系数**：`np.corrcoef(np.arange(window), v)` 对成交量的线性趋势敏感，但成交量本身波动很大，建议用对数成交量或成交量MA比值
6. **ETF流动性未验证**：部分ETF（如159905油气ETF）流动性较差，实盘可能有较大滑点

### 后续改进方向

- **优先级高**：加入止损机制，建议用ATR止损或固定回撤止损（如单个ETF回撤超过8%减仓）
- **优先级高**：修复ETF池，去除重复，补充主要行业，建议扩展到15-20个行业ETF
- **优先级高**：将动量窗口从20日扩展到60-120日，减少噪音
- **优先级中**：加入行业轮动的宏观过滤层（如PMI、利率环境），避免在熊市中频繁轮动
- **优先级中**：测试持仓行业数从3个扩展到5个，提高分散度
- **优先级低**：加入等权 vs 动量加权的对比，验证是否需要按信号强度分配权重
