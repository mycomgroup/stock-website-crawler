# Alpha 到目标权重映射规范

## 目标

这份文档定义：

> **任何选股因子、轮动分数、风险预算结果，最终都必须映射成标准化的 `目标权重表`，才能进入主仓执行层。**

它解决的是 `31_index_enhancement_base.md` 中尚未定稿的关键缺口：

- `alpha` 很强，不代表能直接下单
- 有候选列表，不代表有组合表达
- 有组合表达，不代表满足主仓风险约束

---

## 适用范围

适用于：

- `FFScore / RFScore / 红利质量` 一类股票 alpha
- 行业/主题轮动分数
- `EPO` / `ES风险平价` 一类权重优化输出
- ETF / 多资产配置

不适用于：

- 纯事件驱动逐笔交易
- 分钟级别盘口策略

---

## 标准输出 contract

目标权重表至少包含以下字段：

| 字段 | 含义 |
|------|------|
| `trade_date` | 生效日期 |
| `asset_code` | 标的代码 |
| `asset_name` | 标的名称，可选 |
| `asset_type` | `stock` / `etf` / `bond` / `gold` / `cash` |
| `alpha_score` | 原始 alpha 分数 |
| `alpha_rank` | 排名 |
| `route_scale` | 主路由给出的总仓位系数 |
| `confirm_score` | 二级确认层得分 |
| `risk_scale` | 风险缩放系数 |
| `raw_weight` | 初始权重 |
| `capped_weight` | 约束后权重 |
| `target_weight` | 最终目标权重 |
| `reason_tags` | 例如 `ffscore,rsrs,confirm_pass` |

### 最终要求

- 所有 `target_weight` 之和应满足组合约束
- 现金必须显式表达，不允许隐式遗漏
- 没有 `target_weight` 的策略，不算进入主仓工程层

---

## 标准流程

### Step 1：得到候选池

先经过：
- `04_base_filters.md`
- 合法资产池定义
- 必要的上市时长、成交额、停牌过滤

输出：
- 候选标的列表
- 对应原始 alpha 分数

### Step 2：标准化 alpha 分数

可接受的标准化方式：

1. 排名分数
2. z-score
3. 分组打分
4. 分位数打分

规则：
- 同一组合内必须统一一种分数口径
- 不允许把不同量纲的分数直接硬相加
- 若多因子合成，先归一化再加权

### Step 3：生成 `raw_weight`

默认允许三种主路线：

#### 路线 A：等权 TopN

适用于：
- 简洁基线
- 候选数不大
- 低频股票主仓初版

#### 路线 B：按分数线性映射

示意：

```text
weight_i ∝ max(alpha_score_i, 0)
```

适用于：
- 分数有明确强弱差异
- 希望高分个股获得更高配置

#### 路线 C：优化器输出

例如：
- `25_epo_portfolio.md`
- `27_es_risk_parity.md`

适用于：
- ETF / 多资产
- 权重层本身就是主要价值来源

---

## 必过的四类约束

### 1. 单票上限

默认必须设置单票上限。

原因：
- 防止高分数带来集中爆仓
- 避免所谓 alpha 其实只是单票暴露

### 2. 行业/主题上限

适用于股票主仓。

原因：
- 防止行业抱团伪装成因子有效
- 便于归因时识别真实收益来源

### 3. 现金缓冲

必须显式保留：
- 调仓摩擦
- 保护仓位
- 主路由防守档空余空间

### 4. 最小持仓数 / 最大持仓数

必须明确：
- 不允许极端集中但口头称“组合”
- 不允许持仓过多导致稀释 alpha

---

## 路由、确认、风险三层如何作用到权重

### 1. 主路由决定总仓位上限

`03_state_router` 或 `08_rsrs_timing` 先给出总仓位档位，例如：

| 状态 | `route_scale` |
|------|------|
| 关闭 | 0.0 |
| 防守 | 0.3 |
| 平衡 | 0.5 |
| 正常 | 0.8 |
| 进攻 | 1.0 |

### 2. 确认层决定能否升档 / 恢复

`30_signalmaker_filters.md` 与 `37_signal_confirmation_interface.md` 只负责：
- 放行
- 降档
- 阻止过早恢复

不直接替代 `raw_weight` 生成逻辑。

### 3. 风险层决定缩放系数

`10_volatility_position.md`、`25`、`27` 可输出 `risk_scale`。

示意：

```text
final_total_exposure = route_scale × risk_scale
```

然后再把个股权重按比例缩放。

## 推荐默认映射

### A. 股票低频主仓默认

```text
候选池
→ FFScore / 红利质量打分
→ TopN 或按分数线性分配
→ 单票上限 / 行业上限 / 现金缓冲
→ 乘以 route_scale
→ 再乘 risk_scale
→ 输出 target_weight
```

### B. ETF / 多资产默认

```text
资产池
→ 路由过滤
→ EPO / ES风险平价 / 等权
→ 现金缓冲
→ 输出 target_weight
```

### C. 微盘 / 小市值条件分支

```text
候选池
→ 微盘Alpha / MAC动量
→ 限制持仓数
→ 强化换手与拥挤度约束
→ 输出 target_weight
```

说明：
- 微盘可以用本规范
- 但不应直接照搬股票低频主仓参数

---

## 最低交付物

每次回测至少输出两张表：

### 表 1：候选评分表

| code | alpha_score | rank | pass_filters | tags |
|------|-------------|------|--------------|------|

### 表 2：目标权重表

| code | raw_weight | capped_weight | route_scale | risk_scale | target_weight |
|------|------------|---------------|-------------|------------|---------------|

没有这两张表，就无法判断：
- 收益来自 alpha 还是来自仓位巧合
- 风险来自选股还是来自过度集中

---

## 常见错误

| 错误 | 问题 |
|------|------|
| 只输出买入列表，不输出目标权重 | 不能统一执行 |
| 路由和选股分数混在一起 | 无法归因 |
| 不保留现金缓冲 | 回测与实盘偏差大 |
| 不设单票/行业上限 | 收益失真 |
| 每次手工改权重规则 | 结果不可复现 |

## 结论

以后任何新策略进入统一增强流程时，必须从“我想买什么”升级为：

- **我为什么买这些标的**
- **我按什么规则分配权重**
- **我怎么把它交给统一执行底座**

这就是从研究想法走向主仓工程的关键一步。
