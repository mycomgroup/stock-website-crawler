# 主仓组合装配图

## 目标

这份文档定义 `universal_mechanisms` 的**主仓统一装配流程**。

原则只有一条：

> **任何策略，不论来源是研报复现、聚宽原型、QuantsPlaybook 模块还是新想法，都必须先过一遍这个通用流程，再进入正式验证。**

这样做的目的不是限制策略，而是：

- 加速验证
- 统一口径
- 快速定位短板
- 把“单点策略”改造成“可复用主仓模块”

---

## 一张图看全流程

```text
策略想法 / 研报 / 现成代码
        ↓
[Step 0] 策略分型
        ↓
[Step 1] 基础股票池 / 标的池过滤
        ↓
[Step 2] Alpha层（选股/轮动/因子分数）
        ↓
[Step 3] 主路由（市场状态 / 大盘择时）
        ↓
[Step 4] 二级确认（过滤器 / 投票 / 再进场）
        ↓
[Step 5] 仓位缩放（波动率 / 风险预算）
        ↓
[Step 6] 权重生成（等权 / 指数增强 / EPO / ES风险平价）
        ↓
[Step 7] 执行与调仓（主仓底座）
        ↓
[Step 8] 退出与保护（移动止盈止损）
        ↓
[Step 9] 统一验证与归因
        ↓
通过 → 进入组合层
不通过 → 返回薄弱环节强化
```

---

## Step 0：先做策略分型

不是所有策略都用一套默认模块，先分型再装配。

| 策略类型 | 是否主仓默认支持 | 推荐装配路线 |
|------|------|------|
| 股票低频主仓 | 是 | `31 + 04 + 24/28 + 03 + 30 + 10 + 17 + 07` |
| ETF / 多资产主仓 | 是 | `31 + 08 + 25/27 + 10 + 17 + 07` |
| 微盘 / 小市值 | 条件支持 | `04 + 01 + 26 + 11 + 15 + 29 + 17 + 07` |
| 事件驱动 / 短线 | 不作为主仓默认 | 独立验证后才能接入组合层 |
| 宏观底部 / 极端信号 | 不作为日常默认 | 作为战略辅助开关 |

结论：
- **主仓默认只支持“低频股票主仓”和“ETF/多资产主仓”两条大线。**
- 微盘是强分支，不应直接覆盖全局默认。
- 事件驱动、竞价、短线机制只能作为卫星仓或机会仓。

## Step 1：基础过滤是强制入口

### 默认必过
- `04_base_filters.md`

### 作用
- 统一排除 ST / 停牌 / 次新 / 科创 / 北交所异常样本
- 统一标的池质量
- 避免后续所有验证被脏样本污染

### 规则
- 新策略若不能接 `04`，默认不得进入主仓验证
- ETF / 多资产策略可跳过股票池过滤，但必须定义合法资产池

---

## Step 2：Alpha层只负责“选什么”，不负责“何时满仓”

### 股票主仓默认 Alpha
1. `24_fscore_selection.md`
   - 默认优先 `FFScore` 路线
2. `28_dividend_quality_filter.md`
   - 防御型、价值型补充 alpha

### 条件 Alpha
- `13_sector_rotation.md`：行业/主题轮动
- `29_mac_momentum_factor.md`：只在微盘/成长支线启用

### 设计原则
- Alpha 只解决“标的优选”
- 不直接承担总仓位开关
- 不允许“选股逻辑 = 仓位逻辑 = 退出逻辑”三者混杂

---

## Step 3：主路由决定“市场给不给做”

### 股票主仓默认主路由
1. `03_state_router.md`
2. `08_rsrs_timing.md`

### 含义分工
- `03`：给出仓位档位（关闭 / 防守 / 平衡 / 正常 / 进攻）
- `08`：做宽基趋势或大盘风险偏好闸门

### 微盘分支主路由
- `26_diffusion_index_timing.md`

### 设计原则
- 主路由负责“做不做、做到几成”
- 不负责具体买哪几只股票
- 主路由应尽量少切换、少误判

---

## Step 4：二级确认层负责降噪，不负责替代主策略

### 默认二级确认层
1. `30_signalmaker_filters.md`
2. `01_emotion_switch.md`

### 条件确认层
- `11_consistency_control.md`：微盘核心风控
- `15_crowding_detection.md`：热点/微盘拥挤修正
- `09_north_money.md`：仅作为辅助确认，不单独主导仓位

### 推荐接法

```text
03状态路由 / 08RSRS 给主判断
        ↓
30SignalMaker 给二级确认
        ↓
01情绪开关做底线兜底
        ↓
确认通过才允许升档或恢复仓位
```

### 设计原则
- 二级确认只能“修正”主路由，不能完全取代主路由
- 所有中等强度信号优先放入这一层，而不是单独立主策略

---

## Step 5：仓位缩放层负责把风险调平

### 默认仓位缩放
- `10_volatility_position.md`

### 条件缩放
- `27_es_risk_parity.md`：多资产组合
- `25_epo_portfolio.md`：多资产 / ETF 权重优化

### 规则
- 股票主仓：先由 `03` 给总仓位，再由 `10` 做波动率缩放
- 多资产主仓：可直接采用 `27` / `25` 输出权重
- 低持仓数股票主仓，不强制默认启用 `25`

---

## Step 6：权重生成层负责把想法变成可执行目标

### 股票主仓默认
- `31_index_enhancement_base.md`

### 推荐映射

```text
FFScore / 红利质量 / RFScore
        ↓
得到候选列表与分数
        ↓
叠加行业上限 / 单票上限 / 现金缓冲
        ↓
生成目标权重表
        ↓
31 主仓底座按月执行
```

### 多资产主仓
- 候选资产池
- 路由与过滤后
- 用 `25` 或 `27` 形成目标权重
- 再交给统一执行层

### 设计原则
- 没有目标权重表的策略，不算进入主仓工程阶段
- “能选股”不等于“能建主仓”

---

## Step 7：执行层必须统一到底座

### 默认执行底座
- `31_index_enhancement_base.md`

### 执行要求
- 月频或明确定义的固定再平衡频率
- 统一基准
- 避免未来函数
- 真实价格
- 明确手续费与滑点
- 统一下单口径

### 目的
- 让所有主仓策略在同一执行框架里对比
- 避免“策略好不好”被“执行写法不同”污染

---

## Step 8：退出层是默认必接模块

### 默认退出层
- `17_trailing_stop.md`

### 参考模板
- `06_exit_rules.md`

### 默认退出规则建议
- 基础止损
- 移动止盈
- 时间止损
- 再进场前必须重新通过主路由 + 二级确认

### 规则
- 没有退出层的策略，默认不能进入主仓实盘候选

---

## Step 9：统一验证与强化

### 当前默认验证模块
- `07_attribution.md`
- `33_master_validation_pipeline.md`
- `36_data_benchmark_cost_spec.md`
- `38_strategy_admission_oos.md`

### 必须统一输出的验证指标

| 类别 | 必要指标 |
|------|------|
| 收益 | 年化收益、累计收益、Alpha、超额收益 |
| 风险 | 最大回撤、Sharpe、Calmar、Sortino、回撤恢复期 |
| 交易 | 胜率、交易次数、平均持有期、换手率、盈亏比 |
| 成本 | 滑点后收益、手续费后收益、压力成本敏感性 |
| 稳定性 | 样本内/样本外、年度分解、市场状态分解 |
| 归因 | 市值、行业、风格、特异收益 |

### 强化流程

```text
1. 若收益不足 → 回看 Alpha 层
2. 若回撤过大 → 回看主路由 / 仓位缩放 / 退出层
3. 若交易过多 → 回看二级确认 / 再进场规则
4. 若收益来源单一 → 回看归因与分散化
5. 若样本外失效 → 回看数据代理与过拟合问题
6. 若费后崩塌 → 回看执行层 / 权重映射 / 成本口径
```

### 验证完成后必须再跑一次统一增强

第一版回测跑完，不允许直接结束。

默认要继续按以下顺序补跑：

```text
36 统一数据 / 基准 / 成本口径
→ 37 确认层 / 双确认 / 再进场
→ 34 Alpha 到目标权重映射
→ 31 指数增强底座统一执行
→ 33 输出 V0 / V1 / V2 / V3 对照报告
→ 38 给出主仓 / 过滤器 / OOS / 淘汰定档
```

这一步就是“新策略验证完成后，再走一遍统一增强”的正式入口。

## 默认主仓骨架

### A. 股票低频主仓默认骨架

```text
04 基础过滤
→ 24 FFScore / 28 红利质量
→ 03 状态路由
→ 08 RSRS
→ 30 SignalMaker确认层
→ 10 波动率仓位缩放
→ 31 指数增强底座执行
→ 17 移动止盈止损
→ 07 归因与统一验证
```

### B. ETF / 多资产主仓默认骨架

```text
资产池定义
→ 08 RSRS / 03 状态判断
→ 30 二级确认（可选）
→ 25 EPO 或 27 ES风险平价
→ 31 统一执行底座
→ 17 退出保护
→ 07 归因与统一验证
```

### C. 微盘/小市值条件骨架

```text
04 基础过滤
→ 29 MAC动量 / 微盘Alpha
→ 26 扩散指数主路由
→ 01 情绪开关
→ 11 一致性风控
→ 15 拥挤度检测
→ 10/17 风控退出
→ 07 归因与统一验证
```

说明：
- 这条骨架有效，但**不是全局默认主仓骨架**。

## 不建议默认挂载到所有主仓的模块

| 模块 | 原因 |
|------|------|
| `02_pause_mechanism.md` | 更适合短线/机会仓，低频主仓默认慎用 |
| `09_north_money.md` | 单独做择时不够稳，建议放进确认层 |
| `14_auction_signal.md` | 竞价/短线属性太强，不适合主仓默认 |
| `16_macro_event_filter.md` | 事件日历依赖外部维护，当前更像辅助模块 |
| `18_multi_timeframe.md` | 有启发，但默认验证证据不足 |
| `20/21/22/23` | 更适合战略辅助，不适合日常默认流 |

## 新策略接入清单

任何新策略接入前，至少回答以下问题：

1. 这是股票主仓、ETF主仓、微盘分支，还是机会仓？
2. 它的 Alpha 在哪一层？
3. 它的主路由是什么？
4. 它的二级确认是什么？
5. 它的仓位缩放规则是什么？
6. 它如何生成目标权重？
7. 它的退出规则是什么？
8. 它是否按 `36` 固定了数据 / 基准 / 成本口径？
9. 它是否能按 `33` 输出统一验证指标？
10. 它是否已完成 `V0 / V1 / V2 / V3` 增强前后对照？
11. 它最终应归入主仓、过滤器、OOS，还是淘汰？
12. 它失败时应优先回改哪一层？

若以上问题答不完整，说明这条策略还没完成工程化。

---

## 结论

`universal_mechanisms` 接下来最重要的任务，不是继续堆更多分散机制，而是把现有机制收敛成：

- **默认主仓骨架**
- **条件分支骨架**
- **统一验证流程**
- **统一强化回路**

从现在起，任何策略都应先映射到这张装配图，再进入下一轮验证。
