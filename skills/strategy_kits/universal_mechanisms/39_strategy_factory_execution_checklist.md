# 策略工厂 / 通用机制抽取执行清单

## 目标

这份清单服务于一个很具体的目标：

> 把已经沉淀在 `聚宽有价值策略558`、`QuantsPlaybook` 等最终策略资产里的**可复用部分**抽出来，形成一套“研报/想法 -> 策略装配 -> 平台提交”的标准流程。

它不再讨论“平台有没有数据”“回测平台怎么用”，这些已经由：

- `skills/backtest_guide/`
- `stock-backtesting-system/skills/`

承担。

这份文档只回答三件事：

1. 应该从最终策略里抽什么
2. 先抽哪些，怎么并行推进
3. 如何让一个新研报/新想法快速落到通用平台执行

## 二、总原则：抽“方法”，不抽“观点”

### 应该抽出来的

- 可被 2 个以上策略复用的过滤器
- 可被复用的风险规则
- 可被复用的调仓 / 仓位 / 退出逻辑
- 可被复用的验证模板
- 可被复用的权重映射和执行假设

### 不应该急着抽的

- 某个策略独有的 alpha 叙事
- 只在单一策略里生效一次的特例逻辑
- 依赖特定市场阶段、且还没有复用证据的写法
- 为了“看起来整洁”而硬拆出来的碎片函数

### 抽取准入标准

任意机制满足以下 4 条里的 3 条，才进入 `strategy_kits`：

1. 至少能被 2 个策略复用
2. 输入输出清楚
3. 可独立测试
4. 不依赖某个单一策略故事

## 四、从 L3 抽到 L2 的执行顺序

### Phase 0：先建“抽取台账”

先不要急着改代码，先把现有策略资产做成一张台账。

#### 任务 P0-1：盘点策略来源

至少覆盖：

- `聚宽有价值策略558`
- `QuantsPlaybook`
- 已有 `docs/universal_mechanisms/` 中提到的机制来源

#### 任务 P0-2：给每个策略打标签

标签至少包括：

- 策略类型：主仓 / 卫星 / 事件 / ETF / 微盘 / 基本面
- 可复用部件：过滤器 / 风控 / 权重 / 验证 / 执行
- 当前状态：原型 / 已验证 / 可迁移 / 待淘汰

#### 交付物

- `strategy_inventory.csv`
- `reusable_parts_catalog.csv`

---

### Phase 1：先抽“最稳定、最容易复用”的东西

第一批不抽 alpha 叙事，先抽机制。

#### 路线 A：基础过滤器

优先级：P0

任务：

1. 统一股票池过滤字段
2. 抽出可配置过滤规则
3. 固化输入输出格式
4. 给出默认过滤组合

#### 路线 B：风控规则

优先级：P0

任务：

1. 抽出空仓规则
2. 抽出止损规则
3. 抽出退出保护
4. 形成统一风险事件日志

#### 路线 C：验证模板

优先级：P0

任务：

1. 固化基线对照矩阵
2. 固化 V0/V1/V2/V3 版本对照
3. 固化成本压力测试模板
4. 固化样本外模板

#### 本阶段目标

先让项目具备：

- 能抽机制
- 能复用机制
- 能统一验证机制

而不是先做很重的“通用策略框架”。

---

### Phase 2：再抽“会明显提升研发效率”的装配件

#### 路线 D：信号装配件

优先级：P1

任务：

1. 把常见 alpha 打分逻辑标准化
2. 把事件增强逻辑模板化
3. 把 `extend_hold` 升级成衰减持仓模板
4. 给信号输出统一 `score/confidence/meta`

#### 路线 E：权重与组合映射

优先级：P1

任务：

1. 统一从分数到目标权重的映射
2. 固化单票上限 / 行业上限 / 现金留存
3. 区分主仓 / 卫星仓权重模板
4. 补齐组合输出表

#### 路线 F：执行假设模板

优先级：P1

任务：

1. 固定成本模板
2. 滑点模板
3. 容量限制模板
4. 部分成交模板

---

### Phase 3：最后才抽“半流程型能力”

#### 路线 G：研报 / 想法快速装配模板

优先级：P2

任务：

1. 设计“策略卡片”输入模板
2. 将策略卡片映射到 `universe / signal / risk / portfolio / validation`
3. 自动生成标准策略目录骨架
4. 自动对接 `skills/` 执行

#### 目标

实现：

```text
研报/想法
-> 策略卡片
-> 选择通用机制
-> 生成策略实例
-> 提交到 skills / 平台回测
-> 输出统一验证报告
```

## 七、策略卡片最小模板

以后任何“研报 / 新想法”先写成一张卡片，再进入装配。

```yaml
strategy_name:
strategy_type: stock_main | etf_main | microcap | event | satellite
source: report | notebook | playbook | legacy_code
market:
benchmark:
rebalance_freq:
asset_universe:
alpha_modules:
risk_modules:
confirmation_modules:
position_modules:
execution_profile:
validation_profile:
notes:
```

### 用途

- 让想法先结构化
- 防止一上来就写平台脚本
- 先决定需要哪些通用机制
- 再决定怎么提交到 `skills/`

## 八、什么东西先不要抽

以下内容默认留在最终策略层：

1. 单一策略独有叙事
2. 还没有复用证据的特殊写法
3. 只为某个平台硬适配的临时代码
4. 只有“看起来像通用”，但其实耦合单一策略的逻辑

判断原则：

- 复用不明确，不抽
- 输入输出不稳定，不抽
- 不能单独验证，不抽

## 九、验收标准

这套执行清单完成后，至少应达到以下状态：

### A. 对策略资产

- 能说清楚每个策略哪些部分属于“方法”，哪些属于“观点”
- 能快速判断哪些值得抽象

### B. 对研发流程

- 新研报 / 新想法能先变成策略卡片
- 策略卡片能快速映射到通用机制
- 通用机制能输出可执行策略骨架

### C. 对平台执行

- 生成的策略实例能直接交给 `skills/` 跑
- 能稳定提交到至少一个通用回测平台

### D. 对策略积累

- 最终策略层更轻
- 通用机制层更强
- 新策略试错速度显著提升

## 结论

你接下来的重点不是继续补“平台说明”，而是：

- 把最终策略资产里的通用机制抽出来
- 把这些机制组织成 `strategy_kits`
- 让新研报 / 新想法可以快速装配
- 再把装好的策略交给 `skills/` 去平台执行

这一步做成之后，项目就不再只是“有很多策略”，而是开始具备真正的**策略工厂能力**。
