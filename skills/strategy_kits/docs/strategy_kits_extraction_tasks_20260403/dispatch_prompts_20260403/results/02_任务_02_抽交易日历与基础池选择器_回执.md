## 最关键的 4 个统一接口

### 交易日历接口（2 个）

1. **`get_trade_days(start, end, market="SSE", count=None)`**
   - 对应聚宽 `get_trade_days`、QP `get_trade_days`
   - 返回 `pd.DatetimeIndex`

2. **`shift_trade_day(date, n, market="SSE")`**
   - 对应 QP `Tdaysoffset`
   - 正数向后、负数向前

### 调仓日接口（1 个）

3. **`get_rebalance_dates(start, end, freq="1M", anchor="last", market="SSE")`**
   - 支持周/月/季/年频 + 多种锚点
   - 返回已对齐到实际交易日的 `pd.DatetimeIndex`

### 基础池接口（1 个）

4. **`resolve_base_universe(universe_type, date, config, adapter)`**
   - 支持 `all_a` / `index` / `etf` / `industry` / `custom`
   - 返回 `List[str]` 代码列表

## 最容易重复建设的地方（3 个高风险点）

### 风险 1：各策略重复写 "去掉 688 / 过滤 ST / IPO 天数"

**现象：** 聚宽策略中几乎每个 `get_universe` 函数都重复这几行。

**预防：** 这部分由任务 01（股票池过滤器）负责，本层只暴露原始基础池，**不重叠**。

---

### 风险 2：调仓日逻辑耦合在平台调度器里

**现象：** 聚宽 `run_monthly(rebalance, 1)` 隐式决定调仓日，策略无法独立拿到日期序列做离线分析。

**预防：** 统一用 `get_rebalance_dates()` 生成纯日期序列，运行底座消费该序列决定何时触发 `rebalance`。

---

### 风险 3：多平台迁移时代码大面积重写

**现象：** 聚宽用 `get_all_securities`，米筐也用 `get_all_securities` 但签名不同，指数成分股 API 更是各不相同。

**预防：** 通过 `BaseUniverseAdapter` 隔离平台差异，新增平台只需增加一个 adapter 文件，策略层代码零改动。
