# API 文档全量抓取与本地存储优化指南

> 目标场景：面对“文档页非常多、结构复杂、存在登录/动态渲染/版本切换”的 API 文档网站，稳定抓全量页面并形成可检索、可增量更新的本地知识库。

## 1. 抓取策略优化（先解决“抓全”和“抓稳”）

### 1.1 链路发现：从“入口 URL”升级为“文档地图”
- 同时使用以下来源构建待抓队列：
  - 菜单/侧边栏导航树
  - 页面内目录（TOC）
  - 搜索接口返回结果（若可用）
  - sitemap、openapi/swagger 索引、版本索引页
- 为每个 URL 标注来源（menu/search/sitemap/manual），便于回溯漏抓原因。

### 1.2 抓取调度：分层优先级 + 并发控制
- 优先级建议：
  1. 版本目录页 / 模块目录页
  2. endpoint 详情页
  3. 示例/FAQ/错误码页
- 使用“域名级并发 + 页面级并发”双限流，避免被封禁或触发风控。
- 引入自适应重试（指数退避 + 抖动），单页失败不阻塞全局。

### 1.3 动态内容：完整触发再抽取
- 对于 SPA 文档站，抓取时执行：
  - 等待关键选择器出现（标题 + 主内容容器）
  - 自动展开折叠块（参数说明、响应示例）
  - 切换所有 tab（Request/Response/Example）
  - 触发分页/“加载更多”
- 对隐藏内容记录 `interaction_trace`，保证可复现。

### 1.4 反爬与稳定性
- 控制 UA、请求速率、会话复用（cookie/profile）
- 对 401/403/429 分类型处理（重登、降速、暂停）
- 保留失败快照（HTML + screenshot + console/network 摘要）用于补抓

## 2. 数据模型优化（先存“原始事实”，再做结构化）

### 2.1 建议的三层存储
1. **Raw 层**：原始 HTML、关键 API 响应、抓取元数据
2. **Normalized 层**：解析后的文档对象（标题、路径、版本、模块、参数、示例）
3. **Index 层**：全文索引 + 向量索引 + 结构化字段索引

### 2.2 文档主键与去重
- 主键建议：`site + version + canonical_path + hash(content_core)`
- 去重维度：
  - URL 去重（规范化后）
  - 内容去重（simhash/minhash）
  - 语义近重复（embedding 相似度）

### 2.3 增量更新策略
- 维护 `content_hash`、`last_seen_at`、`last_modified`
- 每日/每周增量抓取：
  - 未变化：仅更新 `last_seen_at`
  - 内容变化：写新版本并建立 diff
- 对“下线页面”做软删除标记，不立即硬删

## 3. 面向 API 文档的解析增强（提取可用字段）

### 3.1 Endpoint 结构化抽取
至少抽取以下字段：
- HTTP 方法、路径、标题、所属模块
- Query/Header/Path/Body 参数（类型、是否必填、默认值、说明）
- 响应码、响应 schema、错误码
- SDK 示例（curl/python/js 等）

### 3.2 多版本与多语言
- 将 `version`、`lang` 作为一级分区字段
- 同一 endpoint 不同版本建立“演化关系”（新增/废弃/字段变更）

### 3.3 质量校验规则
- 必填字段缺失率（如 path/method）
- 示例代码完整率
- 链接可达率（站内 dead links）
- 结构化解析成功率（按站点/模块统计）

## 4. 检索与应用优化（让本地库真正可用）

### 4.1 混合检索
- 关键词检索：适合字段精确命中（endpoint 名、参数名、错误码）
- 向量检索：适合自然语言问答（“如何分页拉取财务指标”）
- 结构化过滤：`site/version/module/method/tag`

### 4.2 切片策略
- 页面级 + endpoint 级双粒度切片
- 保留“上下文锚点”：面包屑、目录路径、小节标题
- 切片中附带 `source_url`、`captured_at`、`version`

### 4.3 评估体系（建议每周跑）
- Top-K 召回率（真实问题集）
- 答案可溯源率（是否含来源 URL 与版本）
- 过期内容占比（引用旧版本）

## 5. 工程与成本优化

### 5.1 任务编排
- 拆分为独立任务：`discover -> crawl -> parse -> index -> evaluate`
- 每一步可重试、可回放，避免整批失败重跑

### 5.2 可观测性
- 指标建议：
  - 抓取成功率、平均页面耗时、重试率
  - 解析成功率、字段缺失率
  - 索引延迟、检索命中率
- 异常告警：连续 429、登录失效、解析成功率骤降

### 5.3 成本控制
- 热门文档高频增量，低频页面低频刷新
- 对超长页面分块抓取与延迟解析
- 历史快照分层存储（热/冷）

## 6. 在本项目中的落地建议（结合当前 crawler 能力）

你可以基于现有能力逐步升级：
1. 利用当前的链接发现与优先级体系，补充“版本入口页优先”规则。
2. 利用 tab/pagination/dropdown/date-filter 能力，统一沉淀为“交互脚本清单”。
3. 在输出 markdown 的同时保存原始 HTML 与解析 JSON，形成 Raw + Normalized 双轨。
4. 在存储层加入内容哈希与增量调度，避免全量重抓。
5. 增加抓取/解析质量报表（按站点、模块、版本维度）。

## 7. 一个可执行的优化顺序（建议）

- **第 1 周**：补全发现链路 + 抓取稳定性（重试、限流、失败快照）
- **第 2 周**：结构化抽取 endpoint 字段 + 解析质量指标
- **第 3 周**：增量更新与版本 diff
- **第 4 周**：混合检索与评估闭环

---

如果你的目标是“后续做问答/Agent 调用 API”，优先投入：
1) endpoint 结构化抽取质量；2) 版本与时间维度管理；3) 可追溯检索。
