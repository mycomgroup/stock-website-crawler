# 场景识别层设计文档

## 1. 目标

场景识别层的目标不是“预测明天涨跌”，而是将信息层输出的特征转换成**可供策略调度使用的市场状态向量**。

核心输出不是一句主观描述，而是：

**每个时点，市场属于不同 regime 的概率或分数向量。**

例如：

```python
{
    "trend_up": 0.72,
    "rotation_range": 0.18,
    "high_vol_defense": 0.07,
    "liquidity_tight": 0.03,
    "extreme_risk": 0.00
}
```

---

## 2. 设计目标

### 2.1 可解释

每个场景必须能解释为什么得出这个分数。

### 2.2 平滑

场景不能一天一跳，避免策略频繁切换。

### 2.3 可回测

历史任意时点都能复现当时场景分数，不能使用未来信息。

### 2.4 可扩展

先支持规则模型，后续可替换成统计模型或机器学习模型。

---

## 3. 建议的场景定义

第一版建议只保留 5 个顶层场景：

1. 趋势上行 `trend_up`
2. 震荡轮动 `rotation_range`
3. 高波动防守 `high_vol_defense`
4. 流动性收缩 `liquidity_tight`
5. 极端风险 `extreme_risk`

这 5 个场景已经足够支撑大多数股票/ETF/轮动/防守类策略调度。

---

## 4. 场景解释

## 4.1 趋势上行

### 典型特征

- 宽基指数位于中长期均线上方
- 市场宽度改善
- 波动率适中
- 成交额放大但不过热
- 北向或风险偏好资金净流入

### 适合策略

- 趋势跟随
- ETF 动量轮动
- 宽基增强
- 核心资产配置

---

## 4.2 震荡轮动

### 典型特征

- 指数大方向不强
- 行业相对强弱快速变化
- 板块离散度高
- 波动率中等

### 适合策略

- 行业轮动
- 低吸高抛
- 相对强弱切换
- 中性偏轮动类策略

---

## 4.3 高波动防守

### 典型特征

- 波动率明显上升
- 宽度恶化
- 大盘回撤增大
- 风险资产同步走弱

### 适合策略

- 红利低波
- 债券/黄金 ETF
- 防守轮动
- 仓位收缩型策略

---

## 4.4 流动性收缩

### 典型特征

- 成交额持续萎缩
- 融资余额下降
- 利率或信用利差不利
- 高 beta 风格受压制

### 适合策略

- 大盘价值
- 低换手策略
- 低拥挤低波策略

---

## 4.5 极端风险

### 典型特征

- 跌停扩散
- 波动率急升
- 宽度极差
- 情绪快速恶化
- 风险资产共振下跌

### 适合策略

- 现金
- 极低风险避险资产
- 暂停高风险策略

---

## 5. 输入输出设计

## 5.1 输入

来自信息层的标准化特征快照：

- 宏观/流动性得分
- 市场状态得分
- 板块状态得分
- 微观交易得分
- 明细特征
- 质量分

## 5.2 输出

建议输出 3 个对象：

### A. 场景分数向量

```python
regime_scores = {
    "trend_up": 0.72,
    "rotation_range": 0.18,
    "high_vol_defense": 0.07,
    "liquidity_tight": 0.03,
    "extreme_risk": 0.00,
}
```

### B. 主场景标签

```python
primary_regime = "trend_up"
```

### C. 场景解释

```python
reason = [
    "指数趋势强度高于80%历史分位",
    "市场宽度改善",
    "波动率处于中性区间",
]
```

---

## 6. 实现路线

建议分三阶段。

## 6.1 第一阶段：规则打分模型

最适合 MVP。

### 做法

为每个场景定义一组规则：

```python
trend_up_score =
    0.35 * index_trend_strength
  + 0.25 * breadth_score
  + 0.15 * liquidity_score
  - 0.10 * volatility_penalty
  - 0.15 * extreme_risk_penalty
```

### 优点

- 可解释
- 易调试
- 容易回测

### 缺点

- 主观
- 需要人工调参数

---

## 6.2 第二阶段：统计聚类/状态机

基于历史特征做聚类或隐状态识别。

候选方法：

- KMeans + 人工命名
- GMM 高斯混合
- HMM 隐马尔可夫

### 适合用途

- 辅助验证规则场景定义是否合理
- 检查市场状态是否自然分群

---

## 6.3 第三阶段：监督学习场景模型

当有足够历史调度数据和策略表现数据后，再考虑。

目标不是预测指数涨跌，而是预测：

- 哪类策略在未来窗口更适合
- 当前 regime 对组合收益贡献如何

---

## 7. 建议的规则打分框架

## 7.1 特征组分

先从信息层拿到组级得分：

- `macro_liquidity_score`
- `market_trend_score`
- `sector_rotation_score`
- `micro_sentiment_score`

再结合细项特征：

- `market_realized_vol_20d`
- `breadth_ma20_ratio`
- `limit_down_count`
- `turnover_vs_20d_mean`

## 7.2 每个 regime 的评分示意

### 趋势上行

增加项：

- 趋势强
- 宽度强
- 流动性强

扣分项：

- 极端波动
- 跌停扩散

### 震荡轮动

增加项：

- 板块轮动快
- 指数趋势弱但未崩
- 行业离散度高

扣分项：

- 极端风险
- 趋势过强

### 高波动防守

增加项：

- 波动率高
- 宽度弱
- 跌停数多

### 流动性收缩

增加项：

- 成交缩量
- 融资下降
- 北向流出

### 极端风险

增加项：

- 跌停扩散
- 波动飙升
- 宽度崩塌

---

## 8. 平滑与切换机制

场景识别层最怕“信号抖动”。

### 建议方法

#### 8.1 EMA 平滑

对原始场景分数做指数平滑。

#### 8.2 切换阈值

只有当新场景分数显著超过旧场景时才切换主场景。

例如：

- 新场景分数比旧场景高 0.15 以上才切换

#### 8.3 最短持有期

主场景至少维持 3 到 5 个交易日，除非触发极端风险。

#### 8.4 风险优先级覆盖

如果 `extreme_risk` 超过阈值，直接覆盖其他场景。

---

## 9. 与策略调度层的接口

场景识别层不直接决定买卖，而是提供调度依据。

建议输出：

```python
{
    "timestamp": "...",
    "primary_regime": "rotation_range",
    "regime_scores": {...},
    "confidence": 0.68,
    "stability_score": 0.74,
    "reason_codes": [
        "sector_rotation_fast",
        "index_trend_weak",
        "breadth_neutral"
    ]
}
```

调度层再结合策略画像决定：

- 哪些策略允许启动
- 每个策略当前权重上限多少

---

## 10. 如何评估场景模型是否靠谱

不是看“是否预测对市场涨跌”，而是看它是否提升了策略调度质量。

### 建议评估方式

#### 10.1 场景稳定性

- 平均每月切换次数
- 连续维持天数

#### 10.2 场景可解释性

- 每个场景对应特征是否符合直觉

#### 10.3 策略适配度

- 某场景下，被标记适合的策略是否显著更优

#### 10.4 组合提升

- 引入场景调度后，组合收益/回撤/夏普是否改善

---

## 11. 第一阶段 MVP 方案

### 输入

- 10 到 20 个核心日线特征

### 输出

- 5 维场景分数向量
- 主场景标签
- 切换说明

### 方法

- 规则打分 + 平滑 + 阈值切换

### 成功标准

- 能稳定产出场景分数
- 场景解释清晰
- 用于日线策略调度后能改善组合表现

---

## 12. 常见错误

- 把场景识别做成“猜涨跌”
- 场景定义太多，导致无法使用
- 频繁切换，导致策略成本上升
- 直接拿场景分数替代风控
- 使用未来信息

---

## 13. 总结

场景识别层是多策略框架的“大脑前额叶”，它不下交易单，但决定“此时此刻更适合启用哪类策略”。

它的价值不在于预测价格，而在于：

- 降低错配策略上场的概率
- 为仓位分配提供统一依据
- 让多策略系统从“同时跑很多策略”升级为“按环境调度策略”
